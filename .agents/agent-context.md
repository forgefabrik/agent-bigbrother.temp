# Agent Context

## Current System State

Task state is in `.task-locks/registry.json`. Do not edit this file.
It is rebuilt by CI from `TASK_EVENTS.jsonl` via `replayer.mjs` every time
an event is recorded.

To find your task:

```json
// .task-locks/registry.json
{
  "tasks": [
    {
      "task_id": "TASK-0001",
      "status": "TODO",
      "implementation_lock": null,
      "review_lock": null
    }
  ]
}
```

If `implementation_lock` is not null, the task is already claimed.
If `review_lock` is not null, the task is under review.
Do not attempt to claim a locked task.

---

## Valid Task Statuses

All valid statuses are listed in `.task-locks/transitions.yaml` under
`transitions`. Terminal states are listed under `terminal_states`
(`MERGED`, `EXPIRED`).

Do not assume a status exists without checking that file.

---

## Event History

`TASK_EVENTS.jsonl` is an append-only log. Every line is one event.
You never write to this file directly.

You may read it to understand what happened to a task.

Run `node .task-locks/audit.mjs` to verify the chain is intact.

## How Events Are Written â€” Authorized Paths Only

There are exactly two authorized paths for writing to `TASK_EVENTS.jsonl`:

### Path 1: CI workflow (automated)
`task-lock.yml` calls `.task-locks/event-writer.mjs` after your PR is
validated. This is the normal path for task lifecycle events (TASK_CLAIMED,
TASK_HEARTBEAT, TASK_MERGED, etc.).

### Path 2: HTTP gate (programmatic / UI)
`POST /events/write` on the local FastAPI server (`:7337`).
This endpoint validates the payload and calls `event-writer.mjs` as a
subprocess. The UI's `writeTaskEvent()` function uses this path.

```javascript
// From the dashboard (only authorized UI write path):
const result = await writeTaskEvent({
  event_type:     'TASK_HEARTBEAT',
  engine_version: 1,
  timestamp:      new Date().toISOString(),
  task_id:        'TASK-0001',
  agent:          'my-agent',
  role:           'IMPLEMENTATION',
  model:          'gpt-4o',
});
```

### Path 3: Direct CLI (local tooling only)
```bash
echo '{"event_type":"..."}' | node .task-locks/event-writer.mjs --json
```

**In all cases the writer enforces:**
- event_type in the frozen enum (event.schema.json v1.1.0)
- engine_version matches transitions.yaml
- per-type required fields present
- hash chain integrity (event_index, prev_event_hash, event_hash computed fresh)
- advisory write lock (`.task-locks/WRITE.lock`)

No other write path exists. Anything that writes to `TASK_EVENTS.jsonl`
without going through `event-writer.mjs` is a boundary violation.

## Integrity Tools

| Tool | Purpose |
|---|---|
| `audit.mjs` | Verify hash chain, index continuity, schema compliance |
| `replayer.mjs` | Rebuild `registry.json` from events + transitions |
| `snapshot-writer.mjs` | Checkpoint: writes `snapshot_N.json` + `SNAPSHOT_CREATED` event |
| `event-writer.mjs` | ONLY authorized write path for `TASK_EVENTS.jsonl` |

All tools: `node .task-locks/<tool>.mjs --help` for usage, or see `ONBOARDING.md`.
